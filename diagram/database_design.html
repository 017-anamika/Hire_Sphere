Design Rationale

This schema is designed for a balance of performance and scalability, which is critical for a social network application.

  • Users Collection: The central collection. Profile information like experience and education are embedded directly within the user document. This makes loading a user's entire profile page incredibly fast as it requires only one database read.
  • Connections Collection: A separate collection to manage the many-to-many relationships between users. Indexing `requester_id` and `receiver_id` would be necessary to quickly find all connections for a user.
  • Posts, Comments, Likes: These are kept in separate collections because a user can have thousands of them. Referencing them by ID (`author_id`, `post_id`) is more scalable than embedding them.
  • Denormalization: To improve read performance for feeds, fields like `likeCount` are added directly to the `posts` and `comments` collections. These would be updated using database increments whenever a new 'like' document is created.
  • Messaging: The conversation model is efficient. We can quickly find all conversations a user is in by querying the `participants` array. Then, we fetch the messages for a specific conversation.
