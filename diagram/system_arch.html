Design Rationale & Data Flow

This architecture is designed for scalability, performance, and maintainability.

  1. A User interacts with the React App in their browser, which is loaded quickly from a CDN.
  2. For most actions (fetching profiles, creating posts), the React App makes secure REST API calls (HTTPS) to the Node.js API Server through a Load Balancer.
  3. The Node.js server processes the request. It reads/writes user data from/to the primary MongoDB database.
  4. To speed up frequent requests, the server checks for cached data in Redis before querying MongoDB.
  5. When a user uploads a profile picture, the server handles the request and stores the file on Cloudinary, saving only the URL in the database.
  6. For long-running tasks like sending a welcome email, the server pushes a job to a queue in Redis, which is processed by a separate worker, preventing delays for the user.
  7. For real-time chat and notifications, the client maintains a persistent WebSocket connection with the dedicated Socket.IO Server.
